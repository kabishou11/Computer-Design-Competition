# 第一章：比赛选题指南

## 1.1 人工智能赛道概述

### 1.1.1 比赛定位

计算机设计大赛人工智能赛道主要考察：
- **技术创新能力**：新方法、新应用
- **工程实现能力**：完整系统、可运行Demo
- **问题解决能力**：针对实际问题提出方案
- **展示表达能力**：清晰阐述、技术答辩

### 1.1.2 选题原则

```
选题公式 = 技术可行性 × 创新性 × 社会价值 × 展示效果

注意事项：
1. 不要选太难的题目（完不成）
2. 不要选太简单的题目（没亮点）
3. 不要选太偏的题目（评委不懂）
4. 结合自身优势和团队特长
```

## 1.3 不推荐项目类型（挑战类）

### 1.3.1 需要大量计算资源的项目

| 项目类型 | 问题 | 建议 |
|---------|------|------|
| 从零预训练LLM | 需要千卡GPU集群 | 使用开源模型微调 |
| 多模态大模型训练 | 数据和算力要求高 | 基于现有模型封装 |
| 超大规模分布式训练 | 设备和成本要求高 | 使用轻量化方案 |

### 1.3.2 技术风险高的项目

| 项目类型 | 问题 | 建议 |
|---------|------|------|
| 开放式对话系统 | 幻觉问题难解决 | 结合RAG增强 |
| 复杂推理任务 | 准确率难以保证 | 限定应用场景 |
| 实时翻译系统 | 竞品成熟难以超越 | 做垂直领域翻译 |

## 1.4 优秀选题案例

### 1.4.1 教育领域

**项目名称**：AI编程学习助手

**核心功能**：
- 代码自动生成和解释
- 错误定位和修复建议
- 个性化学习路径推荐

**技术方案**：
```
├── LLM (代码生成)
├── RAG (知识库：编程教程)
├── Agent (学习规划)
└── Web界面 (交互)
```

**创新点**：
- 基于学生水平的自适应出题
- 代码错误的智能诊断
- 学习进度可视化

### 1.4.2 医疗健康

**核心功能**：

- 健康知识问答

**技术方案**：
```
├── LLM (对话生成)
├── RAG (医学知识库)
├── 规则引擎 (风险评估)
└── 微信小程序 (交互)
```

**创新点**：
- 个性化健康建议
- 多轮对话健康咨询
- 隐私数据保护

### 1.4.3 企业办公

## 1.5 选题流程

### 1.5.1 需求调研

```
1. 发现问题
   └── 身边的问题、行业的痛点

2. 调研现状
   └── 现有解决方案、竞品分析

3. 确定目标
   └── 解决什么问题、目标用户是谁
```

### 1.5.2 技术选型

```
1. 技术可行性评估
   └── 团队能力、硬件条件、时间周期

2. 技术方案设计
   └── 核心算法、系统架构、数据来源

3. 原型验证
   └── 快速验证核心功能
```

### 1.5.3 选题检查清单

- [ ] 问题是否真实存在？
- [ ] 是否有明确的目标用户？
- [ ] 技术方案是否可行？
- [ ] 团队是否有相关能力？
- [ ] 是否有足够的资源支撑？
- [ ] 项目是否有创新点？
- [ ] 是否能在比赛周期内完成？
- [ ] 是否容易展示和答辩？

## 1.6 资源受限情况下的策略

### 1.6.1 模型选择策略

| 场景 | 推荐方案 | 成本 |
|-----|---------|------|
| 个人电脑 | Qwen2.5-1.5B / Phi-3 | 免费 |
| 有GPU服务器 | Qwen2.5-7B / Llama3-8B | 电费 |
| 无本地GPU | API调用 (通义/Claude) | 按量付费 |
| 完全离线 | llama.cpp + Qwen-7B-GGUF | 免费 |

### 1.6.2 数据策略

- **公开数据集**：利用现有数据集
- **合成数据**：使用LLM生成训练数据
- **小样本学习**：Few-shot prompting
- **知识蒸馏**：从大模型蒸馏到小模型

### 1.6.3 开发策略

- **使用框架**：LangChain、LlamaIndex
- **低代码平台**：Dify、Coze
- **云服务**：薅羊毛使用免费额度
- **开源方案**：利用社区资源
